Project Set 1: Character Sheet

For this project, we will write a program that assists the user in creating their character for a tabletop roleplaying game (TTRPG). This game will be similar to Dungeons and Dragons Fifth Edition (D&D 5e), but many of the rules will be omitted or simplifed.

Background

Dungeons and Dragons is a tabletop roleplaying game. Even if you haven't played a TTRPG before, you've probably heard the name "Dungeons and Dragons" at some point. This assignment will use a modified version of some character creation rules from D&D 5e. The real rules can be found in the 5e System Reference Document (SRD), which you can read if you're curious and want to learn more. However, you do not need to read the SRD for this assignment.

Below is a summary of some terms that will come up in this assignment:

  • A player is a real person playing the game, and their character or player character (pc for short) is the fictional character they control within the game.
  • A character's class is the role they fill in the game's world. In a game like D&D, which focuses on fighting fantasy monsters, a character's class tells us their role in combat. A wizard casts spells, a rogue sneaks up on enemies, and a fighter... fights. Yes, everyone fights, but the fighter fights more directly, like with a sword and shield.
  • A character's ancestry or race (the term varies by game, but I'll usually go with ancestry when not talking about a specific game) represents what kind of creature they are. Standard D&D races include elf, human, dwarf, goblin, gnome, halfling, and orc. However, depending on the game and who's running it, a TTRPG character could be anything: a ghost, a dragon, a talking bear, or a mannequin. I have run a game of D&D that included both a talking bear and a mannequin. The mannequin also talked, not just the bear.
  • A character's HP or hit points represent how difficult they are to defeat in combat.
  • A character's AC or armor class represents how hard they are to hit with an attack. A high AC may mean that a character is very good at avoiding attacks, or that a character is wearing armor that blocks most attacks.
  • A character's ability scores are numbers that represent a character's strength, cunning, speed, and other qualities that affect most of their actions in the game.
  • A character's ability modifiers are bonuses or penalties derived from their ability scores. The modifiers are the numbers we actually use in most calculations, and the scores only exist to determine the modifiers. At this point, you may be wondering why we bother with ability scores if we don't use them after calculating the modifiers. That's an excellent question!
  • A character's level represents how powerful and experienced they are. In most RPGs, characters gain experience and level up to represent how they've grown during their adventures.
  • In addition to standard six-sided dice (referred to as D6 dice), D&D uses several less common dice shapes: D4, D8, D10, D12, and D20.

  • Expanded Project 1.3: Either add more D&D content and new options for stat generation (dice roll and point buy), or redesign the rules and implement your own character creator.

Task 1: Guided Coding

Follow the steps below and write the program as directed. Test your code frequently, at the bare minimum performing every test given in the steps.

If your code is not working as expected, do not move on to the next step until you have fixed it. Moving on without finishing a step correctly will make the problem significantly harder to fix and likely add new problems as you continue working. If you disregard this advice, then you may get to the end of the steps and find it would actually be less work to start over from step 1 than to debug the code you just finished writing.

  1. This program requires two additional pieces of input. Prompt the player for their character's ancestry and class after the character name and before the level. Test these new prompts like you tested the prompts in 1.1: run the program, make sure the prompts appear, and make sure the variables have the expected value after all of the prompts are finished.

  2. If the user enters invalid input, then our program must print an error message and exit. You should modify your prompts to tell the user what input is allowed for the ancestry, class, level, and ability scores. This will be a helpful reminder when you test the program.

    Note: You can check each input value immediately after the prompt, or you can check all of them in a separate section of your program after all of the input is finished. The first option will give a better user experience because you will know right away when you make a mistake, and you won't waste time entering more input before the program exits. It also makes your program faster to test for the same reasons. The second option makes it a little easier to organize the code, and all of the error-checking could be combined into one if statement. I recommend the first option because I believe you'll benefit the most from the ease of testing. My directions will only describe how to code this option.

  3. Add an if statement after the level prompt. A valid level is an integer between 1 and 20, inclusive. We want to check whether it is invalid, so we need a condition that asks whether the level is outside of this range. An invalid level is either less than 1 or greater than 20. There are two parts to this condition, so we need to join them with an AND (&&) or an OR (||). In this case, we should use OR. There are a couple of ways to determine that OR is appropriate: the wording of the condition when we describe it in English only makes sense with the word "or", and we only need to meet one of these two conditions for the level to be invalid (it's not even possible for it to be both less than 1 and greater than 20 at the same time). In the body of our if statement, we need to print an error message and exit our program. The exact error message is ERROR: INVALID INPUT, and we can exit the program with System.exit(1);. Any integer is a valid argument for the System.exit function, but a value other than 0 means the program encountered an error. All together, the prompt with error checking would look something like this:

// prompt for input
System.out.println("level prompt message");
int variableName = nextInt();

// check for invalid input
if (/*TODO: condition*/) {
    println("ERROR: INVALID INPUT");
    // The /exit command won't work here because this whole block is part of a
    // Java code snippet. Jshell expects everything in a snippet to be valid
    // Java code.
    exit(1);
}

  1. You can use a similar prompt to check each ability score. The ability scores must be integers between 8 and 15 inclusive. After you add the error-checking for the level and abilities, you should test each of them a few times. Try a few values inside and outside the range, and make sure the program exits with the error message if and only if you enter a value outside the range. In particular, you should check the boundaries of the range. One of the most common mistakes in this type of code is to make a range inclusive when it should have been exclusive, or to make it exclusive when it should have been inclusive. If you make that mistake, then your code will work correctly for all but one or two numbers. You should check 0, 1, 20, and 21 for the level, as well as 7, 8, 15, and 16 for each ability score.

  2. Once your error-checking code works for the numbers, you also need to check the ancestry and class. Each of these have three options, and the user's input is invalid if it is not equal to one of these options. This means we have three parts to our condition, and they must be joined with AND or OR. Which is appropriate? Let's try describing invalid input more closely to how we would code it (checking each possibility separately) and see whether AND or OR is more natural: an invalid ancestry is not an elf and it is not a dwarf and it is not a human. Therefore we should use AND (&&).

    However, this can be tricky. There's another way to phrase our condition that sounds natural and uses OR instead of AND: "an invalid ancestry is not an elf, dwarf, or human". Does this mean it doesn't matter whether we use AND or OR? No, it still matters. The trick with this phrase is how it looks when structured like code: "it is not true that (an invalid ancestry is an elf or it is a dwarf or it is a human)." We are using or to join several equality checks (instead of inequality checks), then applying the NOT (!) to the entire expression, which I've indicated by wrapping most of the expression in parenthesis.

    You'll develop an intuition for boolean logic with practice, so don't worry if you had a hard time with this part. I'll summarize everything you need to know for the AND solution to this error check:

    • Check whether the ancestry/class is NOT equal to each option.
    • Join each of these comparisons with AND (&&).
    • Never compare strings with s1 == s2 or s1 != s2. This will not work as you want it to. Instead, compare them using the .equals function: s1.equals(s2) for equality and !s1.equals(s2) for inequality.
    • Valid ancestries are "elf", "human", and "dwarf".
    • Valid classes are "fighter", "rogue", and "ranger".
    • Use the lowercase versions of these names. You're free to assume all the user's input for class and ancestry will be lowercase, and I will only use lowercase input for these traits when testing your code. However, if you want your program to ignore case, then you can modify the code that reads their input for class and ancestry by changing nextLine() to nextLine().toLowerCase(). This will convert any capital letters in the user input to lowercase letters. Another solution is to use .equalsIgnoreCase instead of .equals, which will let you check for equality while preserving the user's capitalization. If we use .equalsIgnoreCase and don't convert the input to lowercase, then our output will use the same capitalization as the user. This may or may not be desirable; it depends on how you want the output to look.

    Remember to test your new error-checking code. You should now get an error message if you don't enter a valid ancestry or class, but the program should still work if you enter valid input for everything.

We'll start with adjusting the ability scores. Each ancestry changes one or more ability scores. Humans increase all of their scores by 1, elves increase their dexterity by 2, and dwarves increase their constitution by 2. You can structure this part as an else-if chain:

    // message is for us, not the user. We should only see it if our code is
    // wrong.

This is a separate number from the score, and it's used in most calculations that care about the ability instead of the raw score. The Project 1 Background section explains the difference between ability scores and modifiers in more detail.

The parenthesis aren't necessary, but they emphasize the order of operations. Dividing after subtraction ((abilityScore - 10) / 2) would not give us the correct modifier for some scores below ten.

Valid Input

The first kind of error is more complicated to deal with, and you probably won't learn how to handle those errors until later in the semester.   • Each ability score must be an integer from 8 to 15.

Calculations

This program involves more calculations and variables than the previous program. This section will serve as a reference for all values that are calculated by the program.

  1. Before any other calculations, some of the ability scores must be adjusted by the player's ancestry:
    • If the character is a human, increase each ability score by one.
    • If the character is an elf, increase their dexterity score by two.
    • If the character is a dwarf, increase their constitution score by two.
  2. Each ability score has an associated number called its modifier. The modifier does not alter the score, it is derived from the score. The srd link has a table showing the modifier associated with each score from 1 to 30, but we can calculate the modifier with a formula.
    • Each ability modifier is calculated by dividing the score by 2 and subtracting 5. The formula should look like this: modifier = score / 2 - 5.
    • It's important that you divide before subtracting because integer division will round towards zero, not down. We could perform a similar calculation by subtracting 10 and then dividing by 2, but this would give the wrong value for odd ability scores below 10. For example, a score of 9 would produce a modifier of 0 when it should have a modifier of -1.
  3. The character's HP equals their level multiplied by the sum of their constitution modifier and their "hit die". The hit die is determined by the character's class. A fighter has a hit die of 10, and the other two classes have a hit die of 8. The formula should look like this: hp = level * (conModifier + hitDie).
  4. The character's AC (armor class) uses a different formula depending on their class: - A fighter's AC is equal to 18 - A ranger's AC is equal to 15 plus their dexterity modifier, but their AC cannot be higher than 17 - A rogue's AC is equal to 12 plus their dexterity modifier
  5. The character's proficiency is not displayed, but it is used to calculate their attack bonus. It uses the formula proficiency = 1 + (level + 3) / 4.
  6. Depending on their class, the character has one or two attacks. Each attack has a name and a bonus. The bonus is equal to the character's proficiency bonus plus one of their ability modifiers.
    • A fighter has one attack called Greatsword, which uses their strength modifier
    • A rogue has two attacks: Dagger and Crossbow, which both use their dexterity modifier
    • A ranger has two attacks: Longsword, which uses their strength modifier, and Longbow, which uses their dexterity modifier

Hit Points

A character's hit points can be calculated by rolling their hit die or taking an average value for each level. A hit die is written as "d" followed by the number of sides on the die. For example, "d12" refers to a twelve-sided die. To calculate a character's hit points:

  • Use the maximum value (the number of sides on the die) for first level, regardless of whether you are rolling or taking the average.
  • For each level after first:
    • If rolling, then roll the hit die and add the character's constitution modifier to the result
    • If taking the average, then use 1 + half the die size + the character's constitution modifier
    • A character gains a minimum of 1 hit point per level. If the constitution modifier is negative, then you need to make sure the result for any given level is not less than 1. One way to do this is to use Math.max to take the larger value between 1 and the number you calculated in the previous step.

For example, let's calculate the hit points of a 3rd level fighter with a constitution modifier of -2. In the class table we can see that a fighter's hit die is d10.

  • Rolling (let's assume I roll a 2 and a 7 on a d10)
    • The fighter gains 8 hp at first level: 10 (max hit die value) plus -2 (con)
    • The fighter gains 1 hp at second level: 2 (number rolled on a d10) plus -2 (con) is less than 1, but a character must gain a minimum of 1 hp per level
    • The fighter gains 5 hp at second level: 7 (number rolled on a d10) plus -2 (con)
    • Total hp: 14
  • Average
    • The fighter gains 8 hp at first level: 10 (max hit die value) plus -2 (con)
    • The fighter gains 4 hp at second and third level: 6 (half hit die plus one) plus -2 (con)
    • Total hp: 16

  • version prints the program's version.
  • status checks the current status of each of your projects: whether the expected files/folders were found, whether you've previously tested that project, and whether it is currently passing the tests.
  • test runs the tests for one or more projects.
  • pack zips the files for every guided project that has passed its tests to prepare them for upload to Blackboard.

For example, you can view the help text by running sh tests/test.sh help.

When running this program, make sure you read any error messages it shows you. I've tried to include error messages that cover common problems you may encounter. I expect you to try following any directions in the error messages and using the help command before asking me why something isn't working.

2. Testing a Project

To test a project, run the test command followed by the number of the project you want to test (1.1, 1.2, etc.). You can also run the test command with no project numbers to test all projects that can be found and either haven't been tested or have been modified since their last test. For example, sh tests/test.sh test 1.1 would run the tests for project 1.1, whereas sh tests/test.sh test would attempt to test every project that appears ready for and in need of testing.

If you entered a valid project number, the program will attempt to run all of the tests for that project with your code. This may take a minute or two. After it's finished, you should see a message informing you that the test is complete. You can now check the test results with the following commands:

  • summary displays a summary of the entire set of tests. If your program passed every test case, then the overall result should say PASS. If any tests failed, timed out, or unexpectedly crashed, then the overall result should say FAIL.
    • If a test timed out, then your program may have an error that causes it to run for an extended period of time (such as an infinite loop). You should run your program manually to make sure this isn't the issue.
    • If a test crashed, then there was an unexpected error while the test was running. This could be a bug in the test program, and I recommend rerunning the test to see if it happens again.
  • list lists a summary of each individual test case: the test case's id number, whether that test case passed or failed, and how many lines of the expected output matched your output
  • code shows your code used in the test case
  • show displays information about a specific test case. You can specify what information to show, or let the command print everything it has for that test case. A test case can contain a lot of information, so you may need to scroll up to see all of it.
  • help provides information about the commands you can run while browsing test results.
  • quit stops browsing the test results and moves on to the next project (if you're testing multiple projects) or exits the test program. The details of these test results are not saved, so you won't be able to view them again without rerunning the test.

Interpreting Test Results

A failed test case can contain several useful pieces of information:

  • The input shows what input was provided to your code. This may include two categories of input:
    • Program Input shows what the user typed on the terminal. You can replicate this part of the test input by running your code and typing exactly what you see in this section or using I/O redirection.
    • Function Parameters (Before) shows what values were passed to your function when it was called. This is not applicable to jshell scripts, but compiled Java programs will always be tested by calling one of their functions. You can replicate this part of the input by calling the function being tested and passing identical values as arguments. The (Before) means that this is what the parameters looked like before your code ran. Some parameters may be mutable objects (such as arrays), which makes it possible for your code to modify them. If your code modifies mutable parameters unnecessarily, then it will not pass the test.
  • The student output shows the result of running your code. There are three possible categories included in the output:
    • Program Output shows what the program printed when running.
    • Function Parameters (After) shows what the function parameters looked like after running your code. This only applies to compiled Java tests, and it will only matter when the parameters include mutable objects such as arrays.
    • Function Return Value shows what the function returned at the end of the test. This only applies to compiled Java programs, and it will only show up when testing a function with a return type other than void.
  • The expected output shows the expected results of running code that meets all of the project's requirements. It will include the same categories as the student output. If any of these don't match your output, then the test case will fail.
  • The errors section shows errors that occurred while the test case was running.
  • The additional feedback section may contain more information about the test case or hints about common mistakes that would cause this test case to fail.

When you're looking at the output sections, you'll see each line labelled as "MATCH", "NO MATCH", or "ERROR". These labels are there to help you find the mistakes in your program's output that caused it to fail this test case.

  • MATCH means this line was found in both your output and the expected output. Every line in the expected output must match for the test to pass.
  • NO MATCH means this line of output didn't match the expected output and will therefore be ignored when determining whether the test passed. This is used for your program's output that doesn't match the expected output because your code is allowed to print extra output.
  • ERROR means this line of the output didn't match and caused the test to fail. This is used for all function-related output (parameters after and return value), as well as the expected program output. All of this output must match exactly for your code to pass the test case.

3. Packing and Submitting a Project

After a project passes its tests, you need to put all of your project files in a single zip folder before uploading them to Blackboard. You can do this manually, or you can use the test program's pack command.

With pack Command

Run the test program with the pack command to copy the files for every project that has passed its tests into a zip file. You can then upload this zip to Blackboard. Warning: remember to check the contents of the zip before uploading it and make sure it contains everything you were expecting. My program most likely packed everything correctly, but it's always possible that it made a mistake due to a bug.

As of 0.7.0, the pack command will package projects 1.3 and 2.3 separately from the guided projects. These open-ended projects should be submitted separately. You should see the following zip folders after running pack if you have completed all six projects:

  • guided_project_submission.zip
  • expanded_project_1_3_submission.zip
  • expanded_project_2_3_submission.zip

Submit each zip file on Blackboard using the appropriate link.

Manually Create Zip

If you want to create the zip folder manually, then you need to check that all of your files and folders are named correctly. The status command should take care of this, and it will also let you know whether that project was previously recorded passing its tests. You should be able to select all of your project folders, right-click, and then compress or create a new zip folder (the exact command will depend on your operating system).

You can also use the zip command on the terminal. You'll need to specify the full path to each file this way. For example, the command to zip projects 1.1 and 1.2 would look like this: zip guided_project_submission.zip project1/part1/CharacterSheet1.jsh project1/part2/CharacterSheet2.jsh. The zip utility is not included with Git Bash, so it probably won't be available on Windows.

Be aware that if files or folders are misnamed (either incorrect spelling or incorrect capitalization), then the scripts I use to process submissions will not handle your files correctly and I will not mark them as complete. This is why the test program includes the pack and status commands: to check that everything is ready to submit and zip it correctly. If you don't want to use them, then you need to take care to name and package the files exactly as required before submitting them. Don't create the zip manually to avoid fixing a problem the pack or status command warned you about.
